Lab 2 - HTML Forms and Accessibility

Duration: 2 hours. This lab continues from Lab 1: you keep the same Git and GitHub Pages workflow and add a contact (or feedback) form page to your site. You will use semantic form markup, proper labels, and accessible controls so keyboard and screen-reader users can use the form. See Lecture 1 - Interactivity (forms and focus) and Resources.

On this page: Overview · Lab setup · Building on Lab 1 · Forms and accessibility · Accessibility tasks and examples · Task 1 · Task 2 · Task 3 · Task 4 · Deliverable · Extra · References.

Overview

Objectives

  • Add a second page (contact or feedback form) to your course site with the same semantic structure and document outline as Lab 1
  • Implement an accessible form: explicit labels, appropriate input types, semantic grouping (<fieldset> / <legend>), and native validation (required, pattern)
  • Apply accessibility good practices: skip link, landmark labels (aria-label), optional helper text via aria-describedby, and keyboard-operable layout
  • Verify accessibility by keyboard testing and HTML validation; link the form page from the homepage and publish via your public repo

Concepts

  • Forms: <form>, action, method; <label for="..."> and id; input types (text, email, tel, url); <textarea>; required and pattern; <fieldset> and <legend>; <button type="submit">
  • Accessibility (a11y): Perceivable (labels, no information by colour alone); Operable (keyboard, skip link, focus order); Understandable (grouping, error hints); Robust (valid HTML, semantic structure). ARIA where needed: aria-label, aria-describedby, aria-required (redundant with required but sometimes used).

Tasks (see sections below): accept Lab 2 and clone → build form page with semantic structure → implement accessibility tasks → link from homepage and test with keyboard → push and submit → HTML validation (Task 4).

Deliverable: A contact/feedback form page meeting the accessibility requirements below, linked from your course homepage, committed to the private Lab 2 repo and published via your public repo; HTML validated; live site URL submitted on the course platform.

Extra work [1]: Complete exercises 5-1, 5-3, and 6-1 from the primary textbook (see Resources - Bibliography [1] and Assignments).

Lab setup

Before starting, ensure you have completed Lab 1 and have access to your course homepage, your private Lab 1 repo, and your public GitHub Pages repo. For Lab 2 you will need the same tools: a code editor, Git, and a browser. Use the task order below; the final step is HTML validation (Task 4).

  1. Setup and form structure: Accept Lab 2 assignment, clone repo, create contact.html (or feedback.html) with full document shell (header, nav, main, footer). Add the form with three fieldsets: (1) Your details (name, email, phone optional), (2) Your message (subject/topic, message body), (3) A single question e.g. “How did you hear about us?” with radio buttons (shared name), each with its own <label>. Use correct input types and a submit button. Link the form page from the homepage and the homepage from the form page.

  4. HTML validation (last task): Validate all HTML pages with the W3C validator; fix any errors and push again if needed. Your deliverable must pass validation (see Task 4).

Forms and accessibility

Forms collect input from users. For accessibility and usability, every control must have a clear label, and the form should be keyboard-operable and understandable by screen readers.

Form structure

Use <form> with action (the URL that receives the submission) and method (get or post). Only controls with a name attribute are submitted. For this lab you may use action="#" or a placeholder URL if you are not yet handling submissions on a server.

Labels and control association

Every form control (input, textarea, select) must have a label. The best way is to use <label for="control-id"> and give the control id="control-id". That way clicking the label focuses the control, and screen readers announce the label when the control is focused. Do not use placeholder text as the only label; placeholders are hints, not replacements for visible labels.

Input types

Choose the right type for each field: text, email, tel, url, number, date, etc. Browsers can validate (e.g. email format) and offer better UX (e.g. numeric keyboard on mobile). Use <textarea> for multi-line text (e.g. a message). Use required for mandatory fields and pattern when you need a specific format (e.g. phone).

Grouping with fieldset and legend

Group related controls with <fieldset> and give the group a <legend> (e.g. "Contact details" or "Your message"). This helps screen-reader users understand the form structure. For this lab use three fieldsets: (1) "Your details" (name, email, phone), (2) "Your message" (subject, body), (3) A single question (e.g. "How did you hear about us?") with radio buttons, each with its own <label>.

Focus and buttons

Tab moves focus to the next focusable element; Shift+Tab to the previous. Links, buttons, and form controls are focusable by default. Use <button type="submit"> for the submit action and <button type="button"> for actions that do not submit. Do not use a <table> to lay out the form; use semantic markup and CSS for layout.

Accessibility tasks and examples

Complete the following accessibility requirements on your form page. Each item includes a short rationale and, where useful, example markup.

1. Skip link

Provide a “Skip to main content” link as the first focusable element so keyboard and screen-reader users can bypass repeated navigation. It should be visible when focused (e.g. on Tab) and link to #main-content; the <main> element must have id="main-content".

Example:

<a href="#main-content" class="skip-link">Skip to main content</a>
<!-- nav, header, etc. -->
<main id="main-content">
  <!-- page content -->
</main>

Style the skip link so it is hidden off-screen by default and appears (e.g. top-left) on focus. The course site uses this pattern; you can reuse the same class or inline styles.

2. Explicit labels for every control

Every <input>, <textarea>, and <select> must have an associated <label>. Use <label for="id-of-control"> and set id on the control so that clicking the label focuses the control and assistive technologies announce the label.

Example:

<label for="user-email">Email (required)</label>
<input type="email" id="user-email" name="email" required>

<label for="user-message">Message (required)</label>
<textarea id="user-message" name="message" rows="5" required></textarea>

Do not rely on placeholder text as the only label; placeholders are supplementary hints and may not be announced.

3. Group related controls with fieldset and legend

Use <fieldset> and <legend> to group related fields. Screen readers can announce the group and legend, improving structure. Use three fieldsets: (1) “Your details”, (2) “Your message”, (3) A single question (e.g. “How did you hear about us?”) with radio buttons, each with its own <label>.

Example:

<form method="post" action="#" aria-label="Contact form">
  <fieldset>
    <legend>Your details</legend>
    <label for="name">Full name (required)</label>
    <input type="text" id="name" name="name" required>
    <label for="email">Email (required)</label>
    <input type="email" id="email" name="email" required>
    <label for="phone">Phone (optional)</label>
    <input type="tel" id="phone" name="phone">
  </fieldset>
  <fieldset>
    <legend>Your message</legend>
    <!-- subject, message, etc. -->
  </fieldset>
  <button type="submit">Send message</button>
</form>

4. Landmark labels (aria-label)

Give the form and the navigation a descriptive name so assistive technologies can distinguish them. Use aria-label on <form> and <nav>.

Example:

<nav aria-label="Main navigation">
  <ul><li><a href="index.html">Home</a></li><li><a href="contact.html">Contact</a></li></ul>
</nav>

<form method="post" action="#" aria-label="Contact form">

5. Helper text with aria-describedby

When a field has additional instructions (e.g. “Include your student ID if asking about grades”), associate that text with the control using aria-describedby so screen readers announce it when the control is focused. The helper element must have an id.

Example:

<label for="subject">Subject (required)</label>
<input type="text" id="subject" name="subject" required
       aria-describedby="subject-hint">
<p id="subject-hint" class="form-hint">e.g. Lab 2 question, schedule, grading</p>

6. Required fields and pattern (native validation)

Use the required attribute for mandatory fields so the browser can validate and report errors. Use pattern when you need a specific format (e.g. phone). Ensure error messages or hints are available (browser default or, in later courses, custom messages).

Example (optional pattern for phone):

<label for="phone">Phone (optional, format: digits and spaces)</label>
<input type="tel" id="phone" name="phone"
       pattern="[0-9\s\-+]+" title="Digits, spaces, hyphens, or plus only">

7. Keyboard testing checklist

Before submitting, verify the form with keyboard only (no mouse):

  • Tab order: Press Tab repeatedly. Focus should move in a logical order: skip link → nav links → form controls (in visual order) → submit button. Shift+Tab goes backward.
  • Skip link: On first Tab, focus should go to the skip link; activating it should move focus to the main content (e.g. the form or the page heading).
  • Labels: When focus is on an input or textarea, the label should be announced (test with a screen reader if possible, or confirm each control has a for/id pair).
  • No focus trap: Focus should never get stuck; you should be able to Tab out of the form and reach the footer or other links.
  • Visible focus: The focused element should be clearly visible (browser default outline or your own focus style). Do not remove focus outlines without providing an alternative.

Minimal form example (reference)

Below is a minimal contact form that satisfies the accessibility requirements. Adapt it to your page (add document shell, nav, footer) and extend as required: add phone field and a third fieldset with a question and radio buttons (e.g. “How did you hear about us?”), each radio with its own <label>.

<a href="#main-content" class="skip-link">Skip to main content</a>
<nav aria-label="Main navigation">
  <ul>
    <li><a href="index.html">Home</a></li>
    <li><a href="contact.html">Contact</a></li>
  </ul>
</nav>
<main id="main-content">
  <h1>Contact</h1>
  <section>
    <form method="post" action="#" aria-label="Contact form">
      <fieldset>
        <legend>Your details</legend>
        <label for="name">Full name (required)</label>
        <input type="text" id="name" name="name" required>
        <label for="email">Email (required)</label>
        <input type="email" id="email" name="email" required>
      </fieldset>
      <fieldset>
        <legend>Your message</legend>
        <label for="subject">Subject (required)</label>
        <input type="text" id="subject" name="subject" required
               aria-describedby="subject-hint">
        <p id="subject-hint" class="form-hint">e.g. Lab 2, schedule, grading</p>
        <label for="msg">Message (required)</label>
        <textarea id="msg" name="message" rows="5" required></textarea>
      </fieldset>
      <fieldset>
        <legend>How did you hear about us?</legend>
        <label for="hear-course">Course / university</label>
        <input type="radio" id="hear-course" name="hear" value="course">
        <label for="hear-search">Search / web</label>
        <input type="radio" id="hear-search" name="hear" value="search">
        <label for="hear-other">Other</label>
        <input type="radio" id="hear-other" name="hear" value="other">
      </fieldset>
      <button type="submit">Send message</button>
    </form>
  </section>
</main>

Task 2 - Build the form page and implement accessibility

Create a new HTML page (e.g. contact.html or feedback.html) with the same document structure as your Lab 1 homepage: <header>, <nav>, <main id="main-content">, <footer>. Use one <h1> for the page title and place the form inside <main> in a <section>. Implement all items in Accessibility tasks and examples.

Form content (required)

  • Fieldset 1 – Your details: Full name (text, required), Email (email, required), Phone (tel, optional; optional pattern if you wish).
  • Fieldset 2 – Your message: Subject or topic (text or select, required), Message (textarea, required). For one of these fields, add helper text and associate it with the control using aria-describedby (see accessibility task 5).
  • Fieldset 3 – Single question (required): A third <fieldset> with a single question, e.g. “How did you hear about us?” with radio buttons (<input type="radio"> and shared name), each with its own <label>.
  • Submit: <button type="submit"> with clear text (e.g. "Send message").

Accessibility checklist (all required)

  • Skip link as first focusable element, pointing to #main-content; <main id="main-content"> present.
  • Every form control has an explicit <label for="..."> with matching id on the control.
  • Three <fieldset>s with <legend>: (1) Your details, (2) Your message, (3) A single question (e.g. “How did you hear about us?”) with radio buttons, each with its own <label>.
  • aria-label on <form> (e.g. "Contact form") and on <nav> (e.g. "Main navigation").
  • At least one control with helper text linked via aria-describedby.
  • required on all mandatory fields; appropriate type (email, tel, text).

Linking and navigation

Add a link to the form page in the <nav> of your homepage (index.html). On the form page, include a link back to the homepage in the same nav. Use relative paths (e.g. contact.html, index.html or ./).

Exercise 2.1 - Keyboard and validation

Run both pages through the W3C Markup Validation Service and fix any errors. Then perform the keyboard testing checklist: Tab through the entire form page, activate the skip link, and confirm focus order and visibility. Document any issues you find and fix them. Commit and push to your private repo and to your public repo.

Task 3 - Push and submit

Ensure your public repository contains both index.html (homepage with a nav link to the form page) and the form page (e.g. contact.html). Push any last changes. Open your live GitHub Pages URL and confirm both pages load and the link between them works. Submit the live site URL on the course platform (same as Lab 1: the grader must be able to view the published site). If the instructor also asks for the Lab 2 repository link, submit that as well.

Task 4 - HTML validation

As a final step, all submitted pages must be HTML validated. Run each page through the W3C Markup Validation Service (by URI, file upload, or pasted source). Fix every reported error and warning; then push the corrected files to your repos. Your Lab 2 deliverable is complete only when index.html and your form page (e.g. contact.html) pass validation with no errors. If you discover validation errors after submitting, fix them and push again; the grader may re-check.
